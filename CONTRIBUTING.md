# Сдача практических заданий через GitHub

Все практические задания сдаются через GitHub Pull Request (PR).

Такой подход используется в реальных командах разработки и позволяет:

- научиться работать с Git и GitHub;
- получить опыт Code Review;
- вести историю изменений;
- оформлять свои решения профессионально.

# Шаг 1. Сделайте Fork репозитория

Перейдите в репозиторий с заданиями и нажмите ЗЕЛЕНУЮ кнопку **Fork**.

После этого в вашем аккаунте GitHub появится собственная копия репозитория.

Пример:

Оригинальный репозиторий:

```text
https://github.com/alphaserver777/Linux_education
```

Ваш форк:

```text
https://github.com/ВАШ_ГИТХАБ/Linux_education
```

---

# Шаг 2. Клонируйте форк на компьютер

Скопируйте ссылку на свой форк и выполните:

```bash
git clone git@github.com:ВАШ_ГИТХАБ/Linux_education.git
cd Linux_education
```

Примечание: выбираем клонирование через ssh (чтобы каждый раз не вводить логин и пароль). Но для этого сначала НЕ ЗАБУДЬТЕ сгенерировать и сохранить на GitHub аккаунте свой ssh ключ. Подсказка на 35 строке.

---

# Шаг 3. Создайте ветку для домашнего задания

Для каждого задания создаётся отдельная ветка.

Пример:

```bash
git checkout -b hw-04-files-YOUR_NAME
```

Рекомендуемый формат:

```text
hw-XX-topic-YOUR_NAME
```

Примеры:

```text
hw-01-navigation-YOUR_NAME
hw-02-files-YOUR_NAME
hw-03-permissions-YOUR_NAME
hw-04-links-YOUR_NAME
```

---

# Шаг 4. Выполните задание

Создайте необходимые файлы (если того требует задание) или запишите ответ сразу после самого задания.

Структура решения может выглядеть так:

```text
hw-04/
├── solution.md
├── commands.sh
└── screenshots/
```

---

# Шаг 5. Сохраните изменения

Добавьте файлы в индекс Git:

```bash
git add .
```

Создайте коммит:

```bash
git commit -m "HW-04: работа с файлами и каталогами"
```

Сообщение коммита должно кратко описывать выполненную работу.

---

# Шаг 6. Отправьте ветку на GitHub

```bash
git push origin hw-04-files-YOUR_NAME
```

Примечание - удобнее сделать это через ssh ключ.

```bash
git remote set-url origin git@github.com:ВАШ-РЕПОЗИТОРИЙ/Linux_education.git
```

---

# Шаг 7. Обновление форка перед созданием Pull Request

**Важно:** перед созданием pull request необходимо обновить свой форк, чтобы избежать конфликтов и убедиться, что ваши изменения основаны на последней версии основного репозитория (`upstream`).

### Шаг 1: Добавьте `upstream` (если еще не добавлен)

Это ссылка на оригинальный репозиторий, из которого вы сделали форк.

```bash
git remote add upstream https://github.com/alphaserver777/Linux_education.git
```

Проверьте, что `upstream` успешно добавлен:
```bash
git remote -v
```

### Шаг 2: Получите последние изменения из `upstream`

```bash
git fetch upstream
```

### Шаг 3: Переключитесь на вашу основную ветку

Обычно это `main` или `master`.

```bash
git checkout main
```

### Шаг 4: Синхронизируйте изменения

Вы можете использовать `merge` или `rebase`. `Rebase` делает историю коммитов более чистой.

```bash
# С помощью rebase (рекомендуется)
git rebase upstream/main

# Или с помощью merge
git merge upstream/main
```

### Шаг 5: Отправьте обновленную ветку в свой форк (`origin`)

```bash
git push origin main
```

### Шаг 6: Переключитесь на вашу рабочую ветку и обновите ее

```bash
# Переключитесь на ветку с вашими изменениями
git checkout ваша-ветка

# Примените изменения из обновленной основной ветки
git rebase main
```

Теперь ваша ветка (`ваша-ветка`) содержит все последние обновления из основного репозитория, и вы готовы создавать Pull Request.

---

# Шаг 8. Создайте Pull Request

Перейдите в свой форк на GitHub.

GitHub предложит создать Pull Request автоматически.

Нажмите:

```text
Compare & pull request
```

Заполните описание Pull Request по шаблону.

---

# Шаг 9. Дождитесь проверки

После создания Pull Request преподаватель:

- проверит решение;
- оставит комментарии;
- при необходимости попросит внести исправления.

Если потребуется доработка:

1. Исправьте замечания локально.
2. Сделайте новый коммит.
3. Выполните:

```bash
git push
```

Pull Request обновится автоматически.

---

# Шаг 10. Задание принято

После проверки Pull Request будет одобрен и принят.

Статусы задания:

🟡 Open — ожидает проверки

🔵 Changes Requested — требуется доработка

🟢 Approved — задание принято

---

# Важные правила

✅ Одно задание = одна ветка

✅ Одно задание = один Pull Request

✅ Перед сдачей проверьте решение самостоятельно

✅ Пишите понятные сообщения коммитов

❌ Не работайте в ветке main

❌ Не удаляйте чужие файлы

❌ Не отправляйте решение в личные сообщения вместо Pull Request

---

# Схема работы

```text
Получил задание
 ↓
Fork
 ↓
Clone
 ↓
Branch
 ↓
Решение
 ↓
Commit
 ↓
Push
 ↓
Обновление из upstream (rebase)
 ↓
Pull Request
 ↓
Review
 ↓
Approved
```

Добро пожаловать в рабочий процесс, который используют реальные команды разработки.
