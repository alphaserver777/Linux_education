## Глава 9. Привилегии — практика (от простого к сложному)

Глава 9 у Шотса — про модель доступа Unix: `id`, `chmod` (восьмеричная и символьная формы), `umask`, `chown`/`chgrp`, `su`/`sudo` и специальные биты `setuid`/`setgid`/sticky. Для DevOps это не теория, а ежедневная гигиена: неправильные права на приватный ключ — и `sshd` тебя не пустит; слишком открытый `.env` — и это находка для аудита безопасности; неверный владелец каталога — и приложение не стартует. Разберём по нарастающей.

---

### Задача 1 (простая). Кто я и в каких группах?

**Задание:** ты зашёл на новый сервер под пользователем `deploy`. Узнай свой UID, основную группу и список всех групп. Проверь, состоишь ли ты в группе `docker` (чтобы понять, можешь ли работать с Docker без `sudo`).

**Решение:**

```bash
id
# uid=1001(deploy) gid=1001(deploy) groups=1001(deploy),998(docker),27(sudo)

id -nG              # только имена групп
groups              # то же самое, короче
id -nG | tr ' ' '\n' | grep -x docker   # явная проверка членства
```

**Объяснение.** `id` — первое, что делаешь на незнакомом сервере. Членство в группе `docker` фактически равно root-доступу (демон Docker работает от root), поэтому в аудитах это первое, что проверяют. Членство в `sudo` (на Debian/Ubuntu) или `wheel` (на RHEL) определяет, можешь ли ты повышать привилегии. Информация берётся из `/etc/passwd` (учётка) и `/etc/group` (группы) — это просто текстовые файлы, как и подчёркивает Шотс.

### Задача 2 (простая). Сделать скрипт исполняемым

**Задание:** ты написал `deploy.sh`, но `./deploy.sh` выдаёт `Permission denied`. Сделай его исполняемым — двумя способами (символьным и восьмеричным), и объясни разницу.

**Решение:**

```bash
ls -l deploy.sh
# -rw-r--r-- 1 deploy deploy 412 May 30 12:00 deploy.sh

chmod +x deploy.sh          # символьная: добавить x всем (u, g, o)
# или точечно — только владельцу:
chmod u+x deploy.sh
# или восьмеричной формой целиком:
chmod 755 deploy.sh         # rwx r-x r-x
```

**Объяснение.** `chmod +x` (без указания кого) = `a+x`, то есть добавляет право на выполнение **владельцу, группе и всем остальным** сразу. `chmod u+x` — аккуратнее: даёт `x` только владельцу, не трогая остальных. Символьная форма удобна тем, что меняет один атрибут, не сбрасывая прочие; восьмеричная (`755`) задаёт картину целиком. Для скриптов важно: файл должен быть и читаемым, и исполняемым — интерпретатор сначала читает содержимое (это прямо отмечено в таблице 9.2 у Шотса).

### Задача 3 (база DevOps). Защитить приватный ключ и `.env`

**Задание:** `ssh` отказывается работать с ключом, ругаясь `UNPROTECTED PRIVATE KEY FILE`. Также у тебя есть `.env` с секретами приложения. Выставь корректные права.

**Решение:**

```bash
chmod 700 ~/.ssh             # rwx ------  только владелец входит в каталог
chmod 600 ~/.ssh/id_ed25519  # rw- ------  приватный ключ — только владельцу
chmod 644 ~/.ssh/id_ed25519.pub  # публичный можно читать всем
chmod 600 ~/.ssh/authorized_keys

chmod 600 /opt/app/.env      # секреты — только владельцу
```

**Объяснение.** `sshd` намеренно отказывается использовать приватный ключ, если он доступен группе или миру (`600` обязателен) — это защита от кражи ключа другими пользователями системы. `700` на `~/.ssh` закрывает сам каталог. Этот же принцип `600` распространяется на любой файл с секретами: `.env`, `kubeconfig`, токены, TLS-приватные ключи. Правило простое: **секрет = `600`, владелец — тот, от кого работает процесс**. Открытый `.env` (например, `644`) — типичная находка пентестеров и причина утечек.

### Задача 4 (средняя). Права для статики веб-сервера: каталоги vs файлы

**Задание:** в `/var/www/site` лежат HTML, CSS, картинки и вложенные каталоги. Сейчас права «вразнобой». Нужно: каталоги — `755`, файлы — `644`, рекурсивно. Сделай это **без** ручного перебора, одной парой команд (или одной командой).

**Решение:**

```bash
# Способ 1 — через find (явно и читаемо):
find /var/www/site -type d -exec chmod 755 {} +
find /var/www/site -type f -exec chmod 644 {} +

# Способ 2 — одной командой через "большую X":
chmod -R u=rwX,go=rX /var/www/site
```

**Объяснение.** Нельзя просто сделать `chmod -R 755` — тогда **все файлы** станут исполняемыми, что и небезопасно, и бессмысленно для картинок. Каталогам бит `x` нужен (без него нельзя «войти» в каталог, `cd`), файлам — нет. Поэтому каталоги и файлы обрабатывают раздельно. Изящный трюк — заглавная `X` в `chmod`: она ставит бит выполнения **только каталогам** (и тем файлам, у кого `x` уже стоял хотя бы у кого-то). `find ... -exec ... +` группирует аргументы в один вызов `chmod` — это быстрее, чем `\;` (отдельный процесс на каждый файл) на больших деревьях.

### Задача 5 (средняя). Передать владение файлами сервисному пользователю

**Задание:** ты распаковал релиз приложения от имени `root`, но процесс должен работать под пользователем `app` и группой `app`. Передай владение всем деревом `/opt/app/current` и проверь.

**Решение:**

```bash
chown -R app:app /opt/app/current     # владелец app, группа app, рекурсивно
ls -ld /opt/app/current
# drwxr-xr-x 5 app app 4096 May 30 12:10 /opt/app/current

# Только группу (chgrp) — если владельца менять не нужно:
chgrp -R app /opt/app/current
```

**Объяснение.** Менять владельца может только root, поэтому `chown` почти всегда идёт под `sudo` в деплой-скриптах. Синтаксис `chown user:group` за один проход меняет и владельца, и группу. Это критично: если приложение пытается писать логи или PID-файл в каталог, которым владеет `root`, оно упадёт с `Permission denied`. Классический паттерн деплоя — «распаковали от root → `chown -R app:app` → запустили под `app`». На веб-серверах аналогично используется `www-data` (Debian) или `nginx`/`apache` (RHEL).

### Задача 6 (средняя). umask: понять права по умолчанию

**Задание:** твой деплой-скрипт создаёт файлы, и они получаются с правами `664`/`775`, а нужно строже. Покажи текущую маску, объясни, какие права получит новый файл, и временно установи маску так, чтобы новые файлы создавались как `640`, а каталоги — `750`.

**Решение:**

```bash
umask           # 0002
touch test.txt; ls -l test.txt
# -rw-rw-r--  (664)

umask 0027      # сбросить права для группы (запись) и всё для мира
touch test2.txt; mkdir d2
ls -ld test2.txt d2
# -rw-r-----  test2.txt  (640)
# drwxr-x---  d2         (750)
```

**Объяснение.** `umask` — это **маска снятия** прав, а не назначения. База для файлов — `666`, для каталогов — `777`; из них вычитаются биты, выставленные в маске. При `umask 0027`: файлы → `666 − 027 = 640`, каталоги → `777 − 027 = 750`. Группа теряет запись, мир — всё. На многопользовательских/общих серверах это важно: маска по умолчанию `0002` (Ubuntu) делает файлы доступными группе на запись, а `0022` — только на чтение. В systemd-юнитах для строгого окружения маску задают директивой `UMask=0027`, чтобы создаваемые приложением файлы изначально были закрыты от посторонних.

### Задача 7 (сложная). setgid на общем deploy-каталоге

**Задание:** у вас есть общий каталог `/srv/deploy`, куда выкладывают артефакты несколько инженеров из группы `devs`. Проблема: каждый создаёт файлы со своей личной группой, и коллеги не могут их перезаписать. Настрой каталог так, чтобы **все новые файлы внутри автоматически принадлежали группе `devs`**.

**Решение:**

```bash
chgrp devs /srv/deploy
chmod 2775 /srv/deploy        # ведущая 2 = бит setgid
ls -ld /srv/deploy
# drwxrwsr-x 2 root devs ... /srv/deploy   ← обрати внимание на 's'

# Распространить групповое владение и setgid на уже существующее дерево:
chgrp -R devs /srv/deploy
find /srv/deploy -type d -exec chmod g+s {} +
```

**Объяснение.** Обычно новый файл наследует **основную группу создателя**, а не группу каталога — отсюда хаос на общих папках. Бит **setgid на каталоге** (восьмеричное `2000`, символьно `g+s`) меняет правило: новые файлы и подкаталоги наследуют **группу каталога** (`devs`). В выводе `ls` это видно как `s` в позиции группового `x` (`rws`). Это стандартный приём для CI/CD-зон, общих репозиториев артефактов, shared-каталогов команды — он избавляет от вечных `chgrp` руками после каждой выкладки.

### Задача 8 (сложная). Sticky bit на общей директории загрузок

**Задание:** есть общий каталог `/var/uploads`, куда несколько сервисов и пользователей пишут файлы. Сейчас любой может удалить чужой файл. Сделай так, чтобы **удалять/переименовывать файл мог только его владелец** (как в `/tmp`).

**Решение:**

```bash
chmod 1777 /var/uploads       # ведущая 1 = sticky bit
ls -ld /var/uploads
# drwxrwxrwt 2 root root ... /var/uploads   ← 't' в конце

# Если запись нужна только группе, а не миру:
chgrp shared /var/uploads
chmod 1770 /var/uploads
```

**Объяснение.** По умолчанию право `w` на каталог позволяет **удалять любые** файлы внутри, даже чужие, — потому что удаление файла это операция над каталогом, а не над файлом. **Sticky bit** (восьмеричное `1000`, символьно `+t`) ограничивает удаление: убрать/переименовать файл может только его владелец (или root). Именно так защищён `/tmp` — он `drwxrwxrwt`. Буква `t` в позиции «мирового» `x` — индикатор sticky bit. Для shared-каталогов загрузок, кэшей, временных артефактов это обязательная настройка.

### Задача 9 (сложная, безопасность). Аудит setuid-бинарников

**Задание:** в рамках проверки безопасности найди все файлы в системе с установленным битом **setuid** (потенциальный вектор повышения привилегий) и пойми, почему таких файлов должно быть мало.

**Решение:**

```bash
sudo find / -xdev -type f -perm -4000 -exec ls -l {} \; 2>/dev/null
# -rwsr-xr-x 1 root root ... /usr/bin/sudo
# -rwsr-xr-x 1 root root ... /usr/bin/passwd
# -rwsr-xr-x 1 root root ... /bin/su

# setgid-бинарники:
sudo find / -xdev -type f -perm -2000 -exec ls -l {} \; 2>/dev/null
```

**Объяснение.** Файл с битом **setuid** (`4000`, виден как `rws` у владельца) выполняется с привилегиями **владельца файла, а не запустившего**. Если владелец — `root`, обычный пользователь, запуская такую программу, получает root-привилегии на время её работы. Это легитимно для `passwd` (которому нужно писать в `/etc/shadow`) и `sudo`, но **любой лишний setuid-бинарник — потенциальная дыра**: уязвимость в нём = повышение до root. Поэтому, как пишет Шотс, число таких программ сводят к минимуму. В `find`: `-perm -4000` ищет файлы, где этот бит установлен (минус = «эти биты включены, остальные неважны»), `-xdev` не уходит на другие ФС, `2>/dev/null` глушит ошибки доступа.

### Задача 10 (сложная, автоматизация). sudo без пароля для одной команды

**Задание:** CI-агент работает под пользователем `ci` и должен уметь перезапускать только сервис `myapp` через `systemctl`, **без интерактивного пароля и без полного root-доступа**. Настрой это безопасно.

**Решение:**

```bash
# Создать отдельный файл правил (не редактировать /etc/sudoers напрямую):
sudo visudo -f /etc/sudoers.d/ci-myapp

# Содержимое файла:
ci ALL=(root) NOPASSWD: /usr/bin/systemctl restart myapp, /usr/bin/systemctl status myapp

# Проверить синтаксис и применить:
sudo visudo -c

# Теперь под ci:
sudo systemctl restart myapp     # пароль не спрашивается
sudo systemctl restart sshd      # будет отказано — не в списке
```

**Объяснение.** Дать CI полный `sudo` — антипаттерн: компрометация агента = компрометация хоста. Правильный подход — **принцип наименьших привилегий**: разрешаем `ci` ровно одну-две конкретные команды через `NOPASSWD`, всё остальное под `sudo` запрещено. Файлы кладут в `/etc/sudoers.d/` отдельными кусками (а не правят основной `/etc/sudoers`), и **только** через `visudo` — он проверяет синтаксис перед сохранением, иначе сломанный sudoers может заблокировать доступ к root целиком. Это связывает главу 9 (`sudo`) с реальной практикой CI/CD и инфраструктурной безопасности.

**Связка с DevOps одной строкой по каждому биту:**

| Право/бит                   | Где встречается ежедневно                        |
| --------------------------- | ------------------------------------------------ |
| `600`                       | приватные ключи, `.env`, `kubeconfig`, TLS-ключи |
| `644` / `755`               | статика и каталоги веб-сервера                   |
| `chown -R app:app`          | передача релиза сервисному пользователю          |
| `umask`/`UMask=`            | строгие права создаваемых приложением файлов     |
| setgid (`2xxx`) на каталоге | общие deploy-/артефакт-каталоги команды          |
| sticky (`1xxx`) на каталоге | общие upload/tmp-зоны                            |
| `find -perm -4000`          | аудит безопасности setuid                        |
| `sudoers.d` + `NOPASSWD`    | минимальные привилегии для CI/автоматизации      |
