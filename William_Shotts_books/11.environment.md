Глава 11 — про окружение оболочки: что в нём хранится (переменные оболочки и окружения, псевдонимы, функции), как его смотреть (`set`, `printenv`, `echo $VAR`, `alias`) и как настраивать через файлы запуска. Ключевые идеи, которые автор выделяет: разница между **переменной оболочки** и **переменной окружения** (последняя наследуется дочерними процессами через `export`); различие **login** и **non-login** оболочек и какие файлы они читают (`/etc/profile`, `~/.bash_profile`, `~/.profile` против `~/.bashrc`); как `PATH` определяет поиск команд. Для DevOps это база: переменные окружения — стандартный способ конфигурировать приложения и CI/CD (12-factor app), а `PATH`/`.bashrc` — то, что чинят первым делом, когда «команда не находится» или «скрипт работает руками, но падает в cron».

### Задача 1 (простая). Посмотреть окружение и достать одну переменную

**Задание:** узнай, какая у тебя оболочка, домашний каталог, имя пользователя и текущий `PATH`. Затем выведи всё окружение целиком и найди в нём переменную `LANG`.

**Решение:**

```bash
echo "$SHELL"        # /bin/bash
echo "$HOME"         # /home/deploy
echo "$USER"         # deploy
echo "$PATH"         # /usr/local/bin:/usr/bin:/bin:...

printenv | less      # всё окружение, постранично
printenv LANG        # значение одной переменной
echo "$LANG"         # то же через подстановку параметра
env | grep -i lang   # альтернатива через grep
```

**Объяснение.** `echo $VAR` — самый быстрый способ достать одну переменную (подстановка параметра из главы 7). `printenv` без аргументов выводит **только переменные окружения**, а с аргументом — конкретную. Кавычки вокруг `"$PATH"` обязательны как привычка: если в значении есть пробелы, без кавычек оно «развалится». На незнакомом сервере проверка `SHELL`/`HOME`/`USER`/`PATH` — это ориентирование: понять, под кем ты, где твой дом и где оболочка ищет команды.

### Задача 2 (простая). Переменная оболочки vs переменная окружения

**Задание:** создай переменную `APP_PORT=8080`. Проверь, что её видит текущая оболочка, но **не** видит дочерний процесс. Затем сделай её переменной окружения и убедись, что теперь дочерний процесс её видит.

**Решение:**

```bash
APP_PORT=8080                 # переменная ОБОЛОЧКИ (не экспортирована)
echo "$APP_PORT"              # 8080 — текущая оболочка видит
bash -c 'echo "child: $APP_PORT"'   # child:  ← пусто! дочерний не видит

export APP_PORT               # делаем переменной ОКРУЖЕНИЯ
bash -c 'echo "child: $APP_PORT"'   # child: 8080  ← теперь видит

# можно за один шаг:
export DB_HOST=db.internal
# проверить, экспортирована ли:
export -p | grep APP_PORT
```

**Объяснение.** Это центральное различие главы. **Переменная оболочки** живёт только в текущей оболочке. **`export`** превращает её в **переменную окружения**, которая наследуется всеми дочерними процессами. Запущенное тобой приложение — это дочерний процесс оболочки, поэтому конфигурация через env-переменные работает только если они **экспортированы**. Отсюда классический баг: написал `APP_PORT=8080` в скрипте, запустил программу — а она его «не видит», потому что забыл `export`. В Docker, systemd (`Environment=`), CI всё конфигурирование строится именно на экспортированных переменных.

### Задача 3 (простая). Псевдонимы для частых команд

**Задание:** посмотри список текущих псевдонимов. Создай псевдоним `ll` для подробного листинга и `k` для `kubectl`. Проверь, во что разворачивается псевдоним, и удали один из них.

**Решение:**

```bash
alias                         # список всех псевдонимов
alias ll='ls -lh --color=auto'
alias k='kubectl'
type ll                       # ll is aliased to `ls -lh --color=auto`
ll                            # проверка

unalias k                     # удалить псевдоним
\ls                           # запустить настоящий ls в обход псевдонима (обратный слеш)
```

**Объяснение.** Псевдоним — это просто текстовая замена первого слова команды. `alias` без аргументов показывает все (это единственная часть окружения, которую не выводят `set`/`printenv`, как отмечает Шотс). `type` показывает, чем является имя — псевдонимом, функцией или бинарником. Псевдонимы живут только в текущей сессии — чтобы сохранить навсегда, их кладут в `~/.bashrc` (задача 6). Обратный слеш `\command` запускает оригинал в обход псевдонима — пригождается, когда `ls`/`rm`/`cp` переопределены, а нужно «чистое» поведение.

### Задача 4 (средняя). PATH: почему «команда не найдена» и как починить

**Задание:** ты установил утилиту в `/opt/tools/bin`, но при вызове `mytool` получаешь `command not found`. Разберись, где оболочка ищет команды, добавь каталог в `PATH` на текущую сессию и проверь, какой именно бинарник запускается.

**Решение:**

```bash
echo "$PATH"                  # /usr/local/bin:/usr/bin:/bin
which mytool                  # ничего — нет в PATH
ls /opt/tools/bin/mytool      # файл есть, но каталога нет в PATH

export PATH="$PATH:/opt/tools/bin"   # добавить В КОНЕЦ
# или в начало (приоритет!):
export PATH="/opt/tools/bin:$PATH"

which mytool                  # /opt/tools/bin/mytool
type -a mytool                # все совпадения по PATH (если есть дубликаты)
command -v mytool             # надёжная проверка наличия команды
```

**Объяснение.** `PATH` — список каталогов через двоеточие, в которых оболочка ищет команды слева направо; первый найденный и запускается. Поэтому **порядок важен**: если в системе два бинарника с одним именем (например, две версии `python` или `terraform`), выигрывает тот, чей каталог в `PATH` левее. `which`/`type -a`/`command -v` показывают, что реально запустится. Изменение через `export PATH=...` действует только в текущей сессии — для постоянного эффекта правят файл запуска. Это ровно та проблема, что Шотс описывает с `$HOME/bin`: каталог надо явно добавить в `PATH`, иначе оболочка о нём не знает.

### Задача 5 (средняя). Login vs non-login оболочка — какой файл читается

**Задание:** определи, в каком типе оболочки ты сейчас работаешь. Объясни, почему переменная, добавленная в `~/.bashrc`, видна в интерактивном SSH-терминале, но может быть не видна в скрипте, запущенном по cron или в `ssh server 'команда'`.

**Решение:**

```bash
shopt -q login_shell && echo "login" || echo "non-login"
echo $0                       # -bash (с дефисом) → login shell

# проверка, какие файлы читаются:
ls -la ~/.bash_profile ~/.profile ~/.bashrc /etc/profile

# демонстрация: non-login non-interactive НЕ читает .bashrc:
ssh localhost 'echo $MY_VAR'  # может быть пусто, даже если MY_VAR в .bashrc
```

**Объяснение.** Это самый «коварный» раздел главы для DevOps. **Login-оболочка** (вход по SSH, консоль) читает `/etc/profile` → затем первый найденный из `~/.bash_profile`, `~/.bash_login`, `~/.profile`. **Non-login интерактивная** (новый терминал внутри сессии) читает `~/.bashrc`. А **non-interactive** (скрипт, cron, `ssh host 'cmd'`) по умолчанию **не читает ни тот, ни другой** — отсюда вечная боль: «скрипт работает, когда я его запускаю руками в терминале, но падает в cron с command not found». Причина — `PATH` и переменные из `.bashrc`/`.profile` в cron-окружении отсутствуют. Поэтому в cron-задачах и системных скриптах `PATH` и нужные переменные задают явно, а не полагаются на профиль.

### Задача 6 (средняя). Постоянная настройка окружения через файлы запуска

**Задание:** сделай так, чтобы при каждом входе были доступны: каталог `~/bin` в `PATH`, переменная `EDITOR=vim`, псевдонимы `ll` и `..` (перейти на уровень выше), и история в 10000 команд без дубликатов. Сделай это безопасно и примени без перелогина.

**Решение:**

```bash
cp ~/.bashrc ~/.bashrc.bak    # СНАЧАЛА бэкап (правило Шотса)

cat >> ~/.bashrc <<'EOF'

# --- my customizations (added 2026-05-31) ---
export PATH="$HOME/bin:$PATH"
export EDITOR=vim
export HISTSIZE=10000
export HISTCONTROL=ignoredups
alias ll='ls -lh --color=auto'
alias ..='cd ..'
EOF

source ~/.bashrc              # применить без перелогина
ll                            # проверка
```

**Объяснение.** Шотс настаивает на двух правилах, которые в DevOps спасают карьеру: **делай бэкап перед правкой конфига** (`cp .bashrc .bashrc.bak`) и **комментируй изменения** (через полгода «ты» не вспомнишь, зачем эта строка). `PATH`/`EDITOR` логически должны быть в `~/.bash_profile`/`~/.profile`, но на практике их часто кладут в `~/.bashrc`, который читается почти всегда. Важно: `'EOF'` в кавычках отключает подстановку — `$HOME` и `$PATH` запишутся буквально, а развернутся при каждом входе. **`source ~/.bashrc`** (или `. ~/.bashrc`) перечитывает файл в текущей оболочке — изменения вступают в силу сразу, без перелогина. Это тот же механизм, которым login-оболочка «втягивает» `.bashrc`.

### Задача 7 (сложная). Конфигурирование приложения через env (12-factor)

**Задание:** приложение читает конфигурацию из переменных окружения (`DATABASE_URL`, `LOG_LEVEL`, `APP_PORT`). Нужно: (а) запустить его с этими переменными, не «загрязняя» текущую оболочку; (б) загрузить набор переменных из файла `.env`; (в) запустить с временно изменённым `LOG_LEVEL`, не трогая остальное.

**Решение:**

```bash
# (а) переменные только для одной команды (инлайн перед командой):
DATABASE_URL="postgres://db/app" LOG_LEVEL=info APP_PORT=8080 ./app
# после завершения эти переменные НЕ остаются в оболочке

# (б) загрузить .env в окружение:
set -a                # автоматически экспортировать всё, что определяется
source .env           # или: . ./.env
set +a
printenv | grep -E 'DATABASE_URL|LOG_LEVEL'

# (в) запустить с временным переопределением одной переменной:
LOG_LEVEL=debug ./app # перекрывает значение только для этого запуска
# изолированное окружение целиком:
env -i PATH=/usr/bin DATABASE_URL="postgres://db/app" ./app
```

**Объяснение.** Это прямое применение главы к практике DevOps. **Инлайн-присваивание** `VAR=value command` задаёт переменную **только** для этой команды — она не остаётся в оболочке, что идеально для одноразовых запусков и не «протекает» в другие процессы. Трюк **`set -a` / `set +a`** включает автоэкспорт: всё, что определяется между ними, автоматически становится переменными окружения — так удобно «втянуть» `.env`-файл (внутри пары `key=value`). **`env -i`** запускает команду в **полностью пустом** окружении (только то, что укажешь явно) — это способ добиться воспроизводимости и не зависеть от мусора в текущей сессии. Конфигурирование через env — основа 12-factor приложений, Docker (`-e`/`--env-file`), Kubernetes (`env`/`envFrom`), systemd (`Environment=`/`EnvironmentFile=`).

### Задача 8 (сложная). Починить «грязный» или сломанный PATH

**Задание:** кто-то в `~/.bashrc` написал `export PATH=/opt/custom/bin` (без `$PATH`), и теперь в сессии не работают даже `ls` и `cat` — `command not found`. Восстанови работоспособность, не перелогиниваясь, найди и исправь ошибку.

**Решение:**

```bash
# Аварийно вернуть базовый PATH в текущей сессии:
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
ls                            # снова работает

# Найти проблемную строку (полными путями, т.к. PATH мог быть сломан):
/bin/grep -n 'PATH=' ~/.bashrc
# 42:export PATH=/opt/custom/bin   ← перезатёр весь PATH!

# Починить — ДОБАВЛЯТЬ к существующему, а не заменять:
/bin/cp ~/.bashrc ~/.bashrc.bak
/bin/sed -i 's#export PATH=/opt/custom/bin#export PATH="$PATH:/opt/custom/bin"#' ~/.bashrc
source ~/.bashrc
echo "$PATH"
```

**Объяснение.** Самая частая и болезненная ошибка с `PATH`: написать `PATH=/opt/custom/bin` вместо `PATH="$PATH:/opt/custom/bin"`. Первое **перезатирает** весь путь — и оболочка теряет `/bin`, `/usr/bin`, после чего перестают находиться даже базовые команды. Спасение: пока ты в сессии, можно вызывать команды **полными путями** (`/bin/ls`, `/bin/grep`) и вручную задать корректный `PATH` через `export`. Правило: при модификации `PATH` **всегда** дописывай к `$PATH`, никогда не присваивай «голое» значение. И всегда тестируй правку `.bashrc` через `source` в текущей сессии, **не закрывая** старый рабочий терминал — иначе можно остаться без рабочего доступа.

### Задача 9 (сложная). Окружение в systemd-сервисе и cron

**Задание:** приложение прекрасно стартует руками в твоём терминале, но падает с `command not found` / «не видит DATABASE_URL» при запуске через systemd-юнит и через cron. Объясни причину и приведи правильную конфигурацию для обоих случаев.

**Решение:**

```ini
# systemd: /etc/systemd/system/myapp.service
[Service]
# у systemd СВОЁ минимальное окружение — твои .bashrc/.profile не читаются!
Environment=LOG_LEVEL=info
Environment=APP_PORT=8080
EnvironmentFile=/etc/myapp/app.env     # переменные из файла KEY=value
ExecStart=/opt/app/bin/app             # полный путь, не полагаться на PATH
```

```bash
sudo systemctl daemon-reload && sudo systemctl restart myapp
```

```bash
# cron: задать PATH и переменные явно в crontab
crontab -e
# наверху crontab:
PATH=/usr/local/bin:/usr/bin:/bin
DATABASE_URL=postgres://db/app
# или подгружать окружение в самой команде:
*/5 * * * * . /etc/myapp/app.env; /opt/app/bin/job.sh >> /var/log/job.log 2>&1
```

**Объяснение.** Это итог главы, обёрнутый в реальную боль продакшена. **Ни systemd, ни cron не читают `~/.bashrc`/`~/.profile`** — у них собственное урезанное окружение с минимальным `PATH` и без твоих env-переменных (см. задачу 5 про non-interactive оболочки). Поэтому «работает у меня в терминале, падает в проде» почти всегда означает: процесс запускается в другом окружении. Решения: в systemd — директивы `Environment=`/`EnvironmentFile=` и **полные пути** в `ExecStart`; в cron — явный `PATH` и переменные в начале crontab или загрузка `.env` прямо в команде. Главный вывод курса по этой главе: **никогда не полагайся на унаследованное окружение в автоматизации — задавай его явно.**

**Связка с DevOps одной строкой:**

| Механизм                      | Где встречается ежедневно                       |
| ----------------------------- | ----------------------------------------------- |
| `export VAR`                  | конфигурация приложений через env (12-factor)   |
| `VAR=val command`             | одноразовый запуск с переменной, без «протечки» |
| `set -a; source .env; set +a` | загрузка `.env`-файла в окружение               |
| `PATH` (порядок!)             | выбор версии бинарника, «command not found»     |
| `~/.bashrc` vs `~/.profile`   | login/non-login, почему cron не видит настроек  |
| `EnvironmentFile=` (systemd)  | конфиг сервиса в проде                          |
| `PATH=` в crontab             | почему скрипт «работает руками, падает в cron»  |
| `source` / `.`                | применить изменения конфига без перелогина      |
