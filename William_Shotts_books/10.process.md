Глава 10 у Шотса — про то, как Linux управляет процессами и как их «приручать»: `ps`, `top`, управление заданиями (`&`, `jobs`, `fg`, `bg`, `Ctrl+Z`/`Ctrl+C`) и сигналы (`kill`, `killall`). Для DevOps это сердцевина работы: «сервис висит», «load average 40», «приложение не отвечает», «процесс не умирает» — всё это глава 10. Ключевая идея, которую Шотс подчёркивает: `kill` не «убивает», а **посылает сигнал**, и от выбора сигнала зависит, успеет ли приложение корректно завершиться. Разберём по нарастающей.

### Задача 1 (простая). Найти процесс приложения и его PID

**Задание:** на сервере крутится Python-приложение. Найди его процесс, узнай PID, владельца, потребление CPU/памяти и от какого пользователя он запущен.

**Решение:**

```bash
ps aux | grep -i python | grep -v grep
# appuser  4821  2.0  3.1 998000 64000 ?  Ssl  10:01  1:12 python3 /opt/app/main.py

ps -ef | grep main.py        # альтернативный «System V»-формат (PPID виден)
pgrep -af main.py            # сразу PID + командная строка, без grep-мусора
pidof python3                # только PID-ы по имени бинарника
```

**Объяснение.** `ps aux` (BSD-стиль, без дефиса — как у Шотса) показывает **все** процессы всех пользователей с колонками `USER`, `%CPU`, `%MEM`, `STAT`, `COMMAND`. `grep -v grep` убирает из вывода сам процесс grep, который иначе попадает в список (классический артефакт). На практике быстрее `pgrep -af` — он ищет по имени/командной строке и сразу даёт чистый PID, который нужен для `kill`, `strace`, `cat /proc/<PID>/...`. Знание PID — отправная точка любой диагностики процесса.

### Задача 2 (простая). Мониторинг в реальном времени через top

**Задание:** сервер «тормозит». Запусти `top`, определи: какая средняя нагрузка, какой процесс ест больше всего CPU и больше всего памяти, сколько зомби-процессов. Отсортируй по памяти и заверши top.

**Решение:**

```bash
top
# top - 14:59 up 6:30, load average: 4.07, 2.02, 1.00
# Tasks: 109 total, 1 running, 106 sleeping, 0 stopped, 2 zombie
```

Внутри `top`:

- `P` — сортировка по `%CPU` (по умолчанию);
- `M` — сортировка по `%MEM`;
- `h` — справка, `q` — выход;
- `k` — послать сигнал процессу прямо из top (введёт PID).

**Объяснение.** Первое, на что смотрят в инциденте, — `load average: 4.07, 2.02, 1.00` (за 1, 5, 15 минут). Это среднее число процессов, конкурирующих за CPU; **ориентир — число ядер**: load 4 на 4-ядерной машине = полная загрузка, на 1-ядерной = перегрузка втрое. Растущий ряд (последнее < первое) = нагрузка нарастает прямо сейчас. Строка `Tasks` показывает `zombie` — мёртвые процессы, не «прибранные» родителем (к ним вернёмся в задаче 7). `top` обновляется каждые 3 секунды — это «живой» взгляд на систему, в отличие от статического снимка `ps`.

### Задача 3 (простая). Запуск длительной задачи в фоне и управление заданием

**Задание:** тебе нужно запустить долгий бэкап (`tar`/`rsync`), но не блокировать терминал. Запусти задачу в фоне, проверь список заданий, верни её на передний план, приостанови и снова отправь в фон.

**Решение:**

```bash
sleep 300 &          # имитация долгой задачи; & = сразу в фон
# [1] 28236          ← [номер задания] PID

jobs                 # список заданий текущей оболочки
# [1]+ Running   sleep 300 &

fg %1                # вернуть на передний план
# (теперь терминал занят)
Ctrl+Z               # приостановить (сигнал TSTP)
# [1]+ Stopped   sleep 300

bg %1                # возобновить, но уже в фоне
# [1]+ sleep 300 &
```

**Объяснение.** `&` запускает команду в фоне, освобождая терминал; оболочка печатает `[1] 28236` — **номер задания** и **PID**. `Ctrl+Z` шлёт сигнал `TSTP` и **приостанавливает** процесс (не завершает!), `bg` возобновляет его в фоне, `fg` — на переднем плане. Спецификатор `%1` ссылается на задание по номеру. На практике это нужно, когда долгая команда «застряла» на переднем плане: `Ctrl+Z` → `bg` спасает, не теряя процесс. Важное ограничение: фоновые задания привязаны к сессии терминала и умрут при разрыве SSH — это решим в задаче 9.

### Задача 4 (средняя). Корректное завершение: TERM против KILL

**Задание:** приложение (PID 4821) надо остановить. Сначала попробуй завершить его «вежливо», дай шанс сохранить состояние; если за несколько секунд не умерло — заверши принудительно.

**Решение:**

```bash
kill 4821             # = kill -TERM (15) — сигнал по умолчанию, мягкий
# подождать пару секунд, проверить:
ps -p 4821            # ещё жив?

kill -TERM 4821       # то же явно
kill -15 4821         # то же номером

# если не реагирует — крайнее средство:
kill -KILL 4821       # = kill -9
kill -9 4821
```

**Объяснение.** Это, наверное, самое важное место главы для DevOps. `kill` по умолчанию шлёт **`SIGTERM` (15)** — «вежливую» просьбу завершиться: приложение может перехватить сигнал, закрыть соединения, дописать буферы на диск, сохранить состояние и выйти чисто. **`SIGKILL` (9)** перехватить нельзя — ядро убивает процесс мгновенно, без шанса «прибраться»: возможны битые файлы, потерянные данные, повисшие блокировки. Правило: **сначала `-TERM`, `-9` — только когда `-TERM` не сработал**. Именно поэтому `systemctl stop` шлёт TERM, ждёт `TimeoutStopSec` и лишь потом KILL — то же самое, но вручную.

### Задача 5 (средняя). Завершить группу процессов по имени

**Задание:** запущено несколько воркеров `gunicorn` (или зависших `python`). Заверши их все по имени, не выписывая PID-ы вручную. Сначала «прорепетируй» безопасно.

**Решение:**

```bash
pgrep -a gunicorn          # СНАЧАЛА посмотреть, кого зацепим
pkill -TERM gunicorn       # послать TERM всем по имени
# или из книги:
killall -TERM gunicorn     # killall работает по имени программы

# по владельцу:
pkill -u appuser python    # все python-процессы пользователя appuser
killall -u appuser

# если не умерли:
pkill -9 gunicorn
```

**Объяснение.** `killall` (из книги) и `pkill` (его современный аналог) шлют сигнал **всем** процессам с подходящим именем — удобно для пулов воркеров, но опасно: можно зацепить лишнее. Поэтому **всегда сначала `pgrep -a`** (сухой прогон — кого затронем), и только потом `pkill`. По умолчанию оба шлют `TERM`, так что сначала мягко, потом `-9`. Чужие процессы сможет завершить только их владелец или root — процессы, как и файлы, имеют владельца (Шотс это подчёркивает).

### Задача 6 (средняя). Перезагрузить конфиг без рестарта — SIGHUP

**Задание:** ты поправил `nginx.conf`. Нужно применить изменения **без рестарта** и обрыва активных соединений.

**Решение:**

```bash
nginx -t                       # сначала проверить синтаксис конфига!
kill -HUP $(cat /var/run/nginx.pid)   # сигнал HUP мастер-процессу = reload
# или:
kill -1 $(pgrep -o nginx)      # -o = самый старый (мастер) процесс
# штатно systemd делает то же самое:
systemctl reload nginx
```

**Объяснение.** Сигнал **`SIGHUP` (1)** исторически означал «обрыв линии терминала», но демоны переопределили его как «**перечитай конфигурацию**». Nginx по HUP плавно перезагружает конфиг: запускает новые воркеры с новыми настройками, старые дорабатывают текущие запросы — **без даунтайма**. Так же ведут себя многие демоны (`sshd`, `haproxy`, `rsyslog`). Ключевая мысль: сигнал — это не всегда «убить»; для демонов HUP — штатный механизм reload. Сигнал шлют именно **мастер-процессу** (`pgrep -o` — самый старый/родительский).

### Задача 7 (сложная). Диагностика зомби и «зависших в D» процессов

**Задание:** в `top` видно `2 zombie`. Найди зомби-процессы и их родителей. Отдельно найди процессы в состоянии `D` (непрерываемый сон) и объясни, почему `kill -9` им может не помочь.

**Решение:**

```bash
# Зомби (STAT = Z):
ps aux | awk '$8 ~ /Z/ {print $2, $11}'    # PID и имя зомби
ps -eo pid,ppid,stat,comm | grep -w Z      # увидеть PPID — родителя

# Процессы в непрерываемом сне (STAT = D):
ps -eo pid,ppid,stat,wchan:25,comm | awk '$3 ~ /D/'
```

**Объяснение.** Колонка `STAT` кодирует состояние: `R` — выполняется/готов, `S` — спит (ждёт события), `D` — **непрерываемый сон** (обычно завис на дисковом/сетевом I/O, например мёртвый NFS), `T` — остановлен (`Ctrl+Z`/STOP), `Z` — **зомби**. Зомби — это процесс, который завершился, но родитель не вызвал `wait()` и не «забрал» его статус; зомби не занимает ресурсов, кроме записи в таблице процессов, и **его нельзя убить — он уже мёртв**. Лечится завершением/перезапуском **родителя** (`PPID`). Процесс в `D` нельзя прибить даже `-9`, пока он не вернётся из системного вызова: `kill` ставит сигнал в очередь, но ядро доставит его только после завершения I/O-операции — поэтому «зависший на диске» процесс иногда не убивается, и это симптом проблем с хранилищем.

### Задача 8 (сложная). Дерево процессов: убить родителя, а не потомка

**Задание:** приложение порождает дочерние процессы (воркеры). Если убить только воркеры — менеджер тут же поднимет новые. Построй дерево процессов, найди корневой (родительский) процесс и заверши именно его.

**Решение:**

```bash
pstree -p appuser            # дерево с PID-ами по пользователю
# main.py(4821)─┬─worker(4830)
#               ├─worker(4831)
#               └─worker(4832)

ps -o pid,ppid,comm --ppid 4821   # все дети процесса 4821

# завершить родителя (он сам корректно остановит детей):
kill -TERM 4821
```

**Объяснение.** `pstree` показывает иерархию «родитель → потомок», которую ведёт ядро (у каждого процесса есть `PPID`). Это критично для пулов воркеров (gunicorn, php-fpm, nginx): мастер-процесс **сам перезапускает** убитых воркеров — поэтому `kill` воркеров бесполезен, они мгновенно возрождаются. Нужно завершать **мастер**, который при получении TERM корректно остановит всю свою группу. Если убить родителя жёстко (`-9`), его дети станут «сиротами» и будут усыновлены `init`/`systemd` (PID 1) — поэтому мягкий TERM мастеру всегда предпочтительнее.

### Задача 9 (сложная). Долгий процесс, переживающий разрыв SSH

**Задание:** нужно запустить долгую миграцию БД по SSH, но соединение может оборваться. Сделай так, чтобы процесс не убился вместе с сессией терминала. Покажи два способа.

**Решение:**

```bash
# Способ 1 — nohup + фон + лог:
nohup ./migrate.sh > migrate.log 2>&1 &
# nohup делает процесс невосприимчивым к SIGHUP при разрыве терминала

# Способ 2 — уже запущенную задачу «отвязать» от сессии:
./migrate.sh &
Ctrl+Z
bg
disown -h %1      # снять задание с управления оболочкой

# Промышленный способ — systemd-юнит или:
systemd-run --user --unit=migrate ./migrate.sh
```

**Объяснение.** Обычное фоновое задание (`&`) привязано к сессии: при разрыве SSH оболочка рассылает потомкам `SIGHUP` (помнишь сигнал из задачи 6?), и они умирают вместе с сессией. **`nohup`** игнорирует HUP, а перенаправление в файл сохраняет вывод. **`disown`** отвязывает уже запущенное задание от таблицы заданий оболочки. Это типичная DevOps-боль: «запустил миграцию, отвалился VPN — миграция оборвалась на середине». Правильнее всего такие вещи запускать не в shell вообще, а через `tmux`/`screen` или systemd-юнит, но `nohup`/`disown` — быстрый рабочий минимум, прямо вытекающий из понимания сигналов главы 10.

### Задача 10 (сложная, инцидент). Полный разбор высокой нагрузки

**Задание:** алерт «load average 12 на 4-ядерном сервере». Проведи диагностику: подтверди нагрузку, пойми, упирается ли система в CPU или в I/O, найди процесс-виновник, заверши его максимально безопасно и зафиксируй вывод для постмортема.

**Решение:**

```bash
uptime                          # load average подтвердить
nproc                           # сколько ядер (контекст для load)

vmstat 2 5                      # 5 снимков с интервалом 2с
#  r  b   ...  us sy id wa
#  если высокий 'r' и низкий id → упор в CPU
#  если высокий 'wa' (I/O wait) и 'b' → упор в диск

# топ-потребители CPU и памяти (статичный снимок для лога):
ps -eo pid,ppid,user,%cpu,%mem,stat,etime,comm --sort=-%cpu | head
ps -eo pid,%mem,rss,comm --sort=-%mem | head

# нашли виновника (PID 9001) — сначала вежливо:
kill -TERM 9001
sleep 5; ps -p 9001 || echo "завершён"
# если жив и реально опасен:
kill -9 9001
```

**Объяснение.** Это сборка всей главы в реальный workflow. `uptime`/`nproc` дают **load в контексте числа ядер** (load 12 на 4 ядрах = трёхкратная перегрузка). `vmstat` отвечает на главный вопрос инцидента: **CPU-bound или I/O-bound?** — колонка `r` (очередь на CPU) против `wa`/`b` (ожидание диска). `ps --sort=-%cpu` сразу выводит виновника наверх и, в отличие от интерактивного `top`, даёт **статичный текст**, который можно приложить к тикету. Завершение — по правилу из задачи 4: TERM → проверка → KILL только при необходимости. `etime` (время жизни процесса) помогает понять, давно ли он «бесится».

**Связка с DevOps одной строкой:**

| Инструмент/сигнал           | Где встречается ежедневно                   |
| --------------------------- | ------------------------------------------- |
| `ps aux` / `pgrep -af`      | найти процесс и PID для диагностики         |
| `top` / `load average`      | первый взгляд в инциденте «сервер тормозит» |
| `vmstat`                    | CPU-bound vs I/O-bound                      |
| `SIGTERM (15)`              | штатная остановка (как `systemctl stop`)    |
| `SIGKILL (9)`               | крайнее средство, риск потери данных        |
| `SIGHUP (1)`                | reload конфига демона без даунтайма         |
| `Ctrl+Z`/`bg`/`fg`/`disown` | спасти/отвязать зависшую задачу             |
| `pstree` / `PPID`           | убивать мастер, а не воркеров               |
| `nohup`                     | пережить разрыв SSH                         |
| `STAT` Z/D                  | зомби и зависшие на I/O процессы            |
