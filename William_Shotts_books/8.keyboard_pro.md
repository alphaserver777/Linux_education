## Глава 8. Разбор

Эта глава у Шотса про то, что отличает «вечного джуна, который печатает всё руками» от инженера, который за секунды достаёт нужную команду из истории и не теряет время на длинные пути. Для DevOps это не «необязательно, но полезно» (как пишет автор) — это базовая скорость работы в инциденте. Ключевое из главы: **Tab-дополнение**, **поиск по истории (`Ctrl+R`)**, **подстановка записей истории (`!`, `!!`, `!$`)** и запись сессии (`script`).

### Задание (сценарий)

Тебя подняли по алерту. Ты зашёл по SSH на сервер `web-01`, где крутится Nginx в Docker. Реши задачи, используя приёмы из главы 8 — **минимум ручного набора**:

1. Ты только что выполнил `cat /var/log/nginx/access.log` и понял, что нужен был `error.log` в том же каталоге. Как переоткрыть файл, не перенабирая длинный путь?
2. Ты набрал `systemctl restart nginx`, получил `Failed: access denied` (забыл `sudo`). Перезапусти одной короткой подстановкой, не набирая команду заново.
3. Создай каталог `/opt/app/releases/2026-05-30` и сразу перейди в него, не копируя путь дважды.
4. Полчаса назад ты запускал длинную команду `docker logs --since 1h --tail 200 nginx_proxy | grep 502`. Найди её в истории и выполни повторно, не вспоминая синтаксис наизусть.
5. Ты выполнил `kubectl get pods -n staging`, а нужен был namespace `production`. Замени одно слово в предыдущей команде максимально коротко.
6. Тебе нужно задокументировать все свои действия по разбору инцидента в файл для постмортема. Как записать весь сеанс терминала?

### Решение и объяснение

**1. Повторное открытие файла — `!$` (последний аргумент).**

```bash
cat /var/log/nginx/access.log
less !$:h/error.log     # !$ = /var/log/nginx/access.log, :h отрезает имя файла
```

Проще и нагляднее — отдельная подстановка последнего аргумента **`Alt+.`** (нажми в командной строке): она вставляет последний аргумент прошлой команды прямо в строку, и ты дописываешь руками:

```bash
less /var/log/nginx/   # набрал less + путь через Alt+. , поправил хвост
```

`!$` — это «последний аргумент предыдущей команды». В реальной работе это самый частый приём: сделал `mkdir foo` → `cd !$`, сделал `vim config.yml` → `cat !$`. Модификатор `:h` (head) отрезает имя файла от пути — полезно, но синтаксис громоздкий, поэтому на практике чаще жмут `Alt+.`.

**2. Перезапуск с sudo — классика `sudo !!`.**

```bash
systemctl restart nginx        # Failed: access denied
sudo !!                         # → sudo systemctl restart nginx
```

`!!` подставляет **всю предыдущую команду** целиком. `sudo !!` — наверное, самый известный one-liner в Linux: ты запускаешь забытую под sudo команду одним движением. bash перед выполнением покажет, что именно подставилось, — это и есть подстановка записей истории из таблицы 8.6 у Шотса.

**3. Создать каталог и зайти — снова `!$`.**

```bash
mkdir -p /opt/app/releases/2026-05-30
cd !$
```

`!$` здесь = `/opt/app/releases/2026-05-30`. Связка `mkdir -p … && cd !$` экономит десятки нажатий при работе с длинными путями релизов/артефактов и исключает опечатку, ведь путь не перенабирается вручную.

**4. Поиск длинной команды — инкрементальный поиск `Ctrl+R`.**
Нажми `Ctrl+R` и начни печатать кусок команды, например `docker logs`:

```
(reverse-i-search)`docker logs`: docker logs --since 1h --tail 200 nginx_proxy | grep 502
```

- `Enter` — выполнить найденное;
- `Ctrl+R` ещё раз — предыдущее совпадение (дальше в прошлое);
- `→` или `Ctrl+J` — вытащить команду в строку для редактирования, не выполняя;
- `Ctrl+G` / `Ctrl+C` — отменить поиск.

Это главный рабочий инструмент главы. Ты не помнишь точный синтаксис флагов `docker logs` — но помнишь, что вчера это работало. `Ctrl+R` + пара символов достаёт рабочую команду из `~/.bash_history`. Альтернатива для аудита — `history | grep docker`, чтобы увидеть номер записи и вызвать `!номер`.

**5. Быстрая замена слова — `^старое^новое^`.**

```bash
kubectl get pods -n staging
^staging^production         # → kubectl get pods -n production
```

Это краткая форма подстановки: «в предыдущей команде заменить первое вхождение `staging` на `production`». Идеально для типичного DevOps-паттерна «та же команда, другое окружение/namespace/хост». Заменяет только **первое** вхождение.

**6. Запись сеанса — `script`.**

```bash
script -a /tmp/incident-2026-05-30.log   # -a = дописывать, не перезатирать
# ... все твои действия по разбору ...
exit                                     # завершает запись
```

`script` пишет в файл весь ввод и вывод терминала — буквально стенограмму сессии. Для постмортема, аудита, обучения или передачи смены это незаменимо: приложил лог к тикету — и видно, что именно делалось на сервере и какой был вывод. Бонус: `script -t` пишет тайминги, потом сессию можно «проиграть» через `scriptreplay`.

---

**Шпаргалка главы 8 для ежедневной работы:**

| Приём                          | Что делает                                 |
| ------------------------------ | ------------------------------------------ |
| `Ctrl+R`                       | поиск по истории (главный инструмент)      |
| `!!` / `sudo !!`               | повтор всей прошлой команды                |
| `!$` / `Alt+.`                 | последний аргумент прошлой команды         |
| `!docker`                      | последняя команда, начинавшаяся с `docker` |
| `^old^new^`                    | замена слова в прошлой команде             |
| `Ctrl+A` / `Ctrl+E`            | в начало / конец строки                    |
| `Ctrl+W` / `Ctrl+U` / `Ctrl+K` | удалить слово слева / до начала / до конца |
| `Tab` (×2)                     | дополнение / список вариантов              |
| `script`                       | запись сеанса в файл                       |
