## Глава 11. Окружение — домашнее задание

Сценарии новые, от простого к сложному. Выполняй на тестовой машине/контейнере, **всегда делай бэкап перед правкой конфигов** и не закрывай рабочий терминал, пока не проверил изменения. Записывай команды и вывод. Решения не прикладываю.

1. Выведи переменные **обоих** видов командой `set` и только переменные окружения командой `printenv`. Найди хотя бы одну переменную, которая есть в выводе `set`, но отсутствует в `printenv`, и объясни, почему так (в чём разница между переменной оболочки и переменной окружения).

set показывает все переменные printenv  только экспортированные

2. Выведи значения переменных `PWD`, `OLDPWD`, `TERM`, `HOSTNAME`. Затем перейди в `/tmp`, вернись командой `cd -` и снова посмотри `PWD` и `OLDPWD`. Как `cd -` связана с переменной `OLDPWD`?

cd использует переменную OLDPWD

3. Создай псевдонимы: `gs` для `git status`, `ports` для просмотра открытых TCP-портов (`ss -tulpn`) и «защитный» псевдоним `rm` → `rm -i`. Проверь через `type`, во что они разворачиваются. Затем запусти **настоящий** `rm` в обход псевдонима. В конце удали псевдоним `rm`.

Alias короткое имя команды Обход alias нужен для запуска настоящей команды

4. Создай переменную `TIMEOUT=30`. Выведи её. Затем **удали** её и покажи, что пустая переменная и несуществующая — это разные вещи. Используя подстановку параметра, сделай так, чтобы команда `echo` вывела значение `TIMEOUT`, а если переменная не задана — значение по умолчанию `10` (подсказка: синтаксис `${VAR:-default}`).

Пустая переменная существует несуществующая нет ${VAR:-10} выводит значение по умолчанию

5. Установи у себя в `/opt/tools/v2/bin` «новую» версию утилиты (можно смоделировать простым скриптом, печатающим `v2`), при этом в `/usr/bin` лежит «старая» (`v1`). Сделай так, чтобы по имени запускалась **новая** версия, не удаляя старую. Подтверди через `type -a` и `which`, какой бинарник реально выигрывает и почему. Что изменится, если добавить каталог в **конец** `PATH`, а не в начало?

Запускается версия из каталога который раньше в PATH Конец PATH имеет меньший приоритет

6. Определи, какой файл запуска использует твой дистрибутив для login-оболочки (`~/.bash_profile`, `~/.bash_login` или `~/.profile`) — проверь, какие из них существуют. Добавь переменную `MY_DEPLOY_ENV=staging` в **правильный** файл (тот, что отвечает за переменные окружения и PATH, а не в `.bashrc`), объясни свой выбор и примени без перелогина.

Переменные окружения хранят в .profile или .bash_profile а не в .bashrc

7. Запусти любую команду (например, скрипт, который печатает `$LOG_LEVEL`) тремя способами: (а) задав `LOG_LEVEL=debug` **инлайн** только для этого запуска; (б) экспортировав переменную и запустив обычно; (в) в **полностью изолированном** окружении через `env -i`. После каждого способа проверь `echo $LOG_LEVEL` в самой оболочке и объясни, в каких случаях переменная «осталась», а в каких — нет.

export сохраняет переменную в окружении env -i запускает программу с пустым окружением

8. (среднесложная) Создай файл `app.env` со строками вида `DATABASE_URL=postgres://db/app`, `LOG_LEVEL=info`, `APP_PORT=8080`. Загрузи все эти переменные в окружение **одной операцией**, чтобы они стали экспортированными (подсказка: `set -a`). Проверь через `printenv`, что приложение-потомок их увидит. Что произойдёт, если просто сделать `source app.env` без `set -a` — будут ли переменные видны дочернему процессу?

set -a автоматически экспортирует переменные Без set -a переменные будут доступны в текущей оболочке, но не будут автоматически экспортированы

9. (сложная) Тебе дали `~/.bashrc`, в котором кто-то прописал тяжёлую инициализацию (например, длинный вывод или запрос к сети) — из-за этого **новые терминалы открываются медленно**, а ещё это ломает `scp`/`sftp`. Разберись, почему вывод в `.bashrc` ломает неинтерактивные сессии, и предложи правку: как сделать так, чтобы «шумные» команды выполнялись **только в интерактивной** оболочке (подсказка: проверка `$-` на флаг `i` или `[[ $- == *i* ]]`).

Вывод в .bashrc может ломать scp и sftp. Его нужно выполнять только в интерактивной оболочке

10. (сложная) Cron-задача `*/5 * * * * backup.sh` падает с `backup.sh: command not found`, хотя из твоего терминала `backup.sh` запускается прекрасно. Объясни причину в терминах login/non-login/non-interactive оболочек и переменной `PATH`. Приведи **два** способа починки: (а) исправление через настройку crontab; (б) исправление внутри самого скрипта, не полагаясь на унаследованное окружение.

Cron использует другой PATH Нужно указать полный путь или настроить PATH
